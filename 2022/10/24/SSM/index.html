Spring IoC

  IoC控制反转：将对象的控制权交由第三方进行控制管理。在生活中的一个例子就是，在我们想去买东西但是不知道东西的具体细节的时候，可以向老板寻求帮助。
  IoC控制反转的核心要义就是并不是由最终的消费者来控制创建对象，取而代之的是通过引入一个中间人的角色，由中间人统一地负责对象的创建过程。老板负责将商品统一地上架售卖，我们不再需要面向商品进行编程，而是面向老板即可，我们与老板之间的会话过程最终由老板统一创建对象。

  1. IoC控制反转,全称Inverse of Control，是一种设计理念
  2. 由代理人来创建与管理对象，消费者通过代理人来获取对象
  3. IoC的目的是降低程序与程序之间直接耦合，通过加入IoC容器将对象统一管理，让对象关联变为弱耦合

 
DI依赖注入：

  1. IoC是设计理念,是现代程序设计遵循的标准,是宏观目标
  2. DI(Dependency Injection)是具体技术实现,是微观实现
  3. DI在Java中利用反射技术实现对象注入(Injection)

 
在传统的开发方式中：

  假设现在A对象想要抛弃B对象，改为关联new一个C对象，此时程序员必须得去修改原代码，将A重新指向C对象，然后重新编译、测试、打包、申请、发布，过于繁琐

在引入Spring IoC容器的开发方式中：

  使用者不需要再去new对象，而是直接从IoC容器中提取所需要的对象，运行机制是在Java的内存中开辟一段空间供Spring IoC容器使用，并且对象统一由Spring IoC容器进行创建
  A创建好了之后，并不是由A去创建B，而是通过反射技术，由A的依赖也就是B对象注入到A对象中，也就是DI，也就是反射技术

Spring AOP

  假设现在有两个软件模块A、B，对于系统中的用户一部分人拥有访问A的权利，一部分人拥有访问B的权利，另一部分人拥有访问两者的权利，那么在书写代码时肯定会想到在A、B之前增加一部分逻辑判断的代码。但是如果有一天不需要这样的权限控制了，程序员就得去找到相应的位置修改代码
  Spring AOP就像是浏览器中的插件，如果使用它来对权限进行控制，不需要的时候直接在配置文件中修改即可，与XML的Spring IoC模式一样，能极大地减少工作量

  1. Spring AOP - Aspect Oriented Programming 面向切面编程
  2. AOP的做法是将通用、与业务无关的功能抽象封装为切面类
  3. 切面可配置在目标方法的执行前、后运行，真正做到即插即用
  4. 可以在不修改源码的情况下对程序功能进行拓展

1. AOP实现原理

  Spring基于代理类实现功能动态扩展,包含两种形式: 如果目标类拥有接口,通过JDK动态代理实现功能扩展；如果目标类没有接口,通过CGLib组件实现功能扩展
  代理就像顾客在租房子的时候，一般很难找到直接房东，因此租客会直接找一些租房平台进行租赁，这些平台就是租房过程中的代理

1.1. 静态代理

  静态代理模式每一次增加新的功能都需要重新手动创建一个新的代理类，十分的麻烦

  代理类和委托类会共同实现一个接口，而客户需要调用接口中的方法，主要的是需要在代理类中有委托类的对象（使用接口类可以实现多级代理）

1.2. 动态代理

  利用反射机制，根据接口的结构，在运行时自动生成代理类。强制要求为：目标类必须实现接口才可以，但是在实际运行时是有挺多接口没有实现的，因此引入了第三方库CGLib组件

1.3. CGLib组件

  1. CGLib是运行时字节码增强技术
  2. Spring AOP扩展无接口类使用CGLib
  3. AOP会运行时生成目标继承类字节码的方式进行行为扩展

  Service没有实现任接口，因此CGLib会在原始目标类的基础上进行继承，然后重写每一个方法来进行增强

CGLib的执行过程：

  1. 生成代理类的二进制字节码文件
  2. 加载二进制字节码，生成Class对象(Class.forName())
  3. 通过反射机制获得实例构造，并创建代理类对象

  1. Spring JDBC是Spring生态用于处理关系型数据库的模块
  2. Spring JDBC对JDBC API进行封装，极大简化开发工作量

为什么有了MyBatis还需要使用Spring JDBC？
  主要原因就是这两者面对的对象是不同的，MyBatis作为一个ORM框架，封装的程度较高，适用于中小型企业进行程序的敏捷开发，让程序员能快速完成与数据的交互工作，但是MyBatis中存在各种XML的配置与操作的细节，各种的操作的封装程度高就意味着执行的效率低
  而Spring JDBC仅仅是对原始的JDBC进行了简单地封装，对于一线大厂无论是数据量还是用户的并发量都是非常高的，因此大都不会选择MyBatis这样的成熟框架，而是会选择使用Spring JDBC这种轻量级框架，然后结合企业的特点进行二次封装

2. 编程式事务

  1. 事务：要么把事情做完，要么什么都不做，不要做一半
  2. 事务依赖于数据库实现,MySQL通过事务区作为数据缓冲地带
  3. 编程式事务是指通过代码手动提交和回滚事务的事务控制方法
  4. SpringJDBC通过Transaction Manager事务管理器实现事务控制
  5. 事务管理器提供commit/rollback方法进行事务提交与回滚

  在没有进行事务管理的时候，如果执行10次插入操作，那么这10个操作是按10个事务来进行的，当有某个操作出错时，前面的操作不会进行撤销。而使用事务管理，能让这10个操作变为一个事务，当有操作出错时，前面执行完在事务区中的操作会进行撤销，整个事务能回滚

3. 声明式事务

  1. 声明式事务指使用配置或注解，在不修改源码情况下自动实现事务。其本质是AOP环绕通知
  2. 当目标方法执行成功时,自动提交事务
  3. 当目标方法抛出运行时异常时,自动事务回滚

3.2. 事务传播行为

  1. 事务传播行为是指多个拥有事务的方法在嵌套调用时的事务控制方式
  2. XML:<tx:method name=”…” propagation=”REQUIRED”/>
  3. 注解:@Transactional(propagation=Propagation.REQUIRED)

事务传播行为七种类型：

事务传播类型 说明
PROPAGATION_REQUIRED(默认) 如果当前没有事务，就新建一个事务，如果已经存在一个事务中，加入到这个事务中。这是最常见的选择
PROPAGATION_SUPPORTS 支持当前事务，如果当前没有事务，就以非事务方式执行
PROPAGATION_MANDATORY 使用当前的事务，如果当前没有事务，就抛出异常
PROPAGATION_REQUIRES_NEW 新建事务，如果当前存在事务，把当前事务挂起
PROPAGATION_NOT_SUPPORTED 以非事务方式执行操作，如果当前存在事务，就把当前事务挂起
PROPAGATION_NEVER 以非事务方式执行，如果当前存在事务，则抛出异常
PROPAGATION_NESTED 如果当前存在事务，则在嵌套事务内执行。如果当前没有事务，则执行与 PROPAGATION_REQUIRED类似的操作

  MVC是一种著名的架构模式，三个单词分别是：Model、View、Controller。View通常指的是界面的部分，用于和用户进行交互，比如通过界面显示查询结果，提供表单接受用户的输入信息；Model通常指的是数据，在Java中通常指业务逻辑部分
  在MVC的架构中，如果视图要调用Model中的数据，并不是直接获得，而是通过控制器Controller这个中介，Controller接收视图中传入的数据，而后再通过传入的数据调用后端的业务逻辑得到结果，再通过控制器将后端的结果返回到视图中
  之前学习的Servlet就是用于开发控制器的技术，但是在使用起来并不是那么的方便，因此Spring机构开发了Spring MVC用于简化Web应用的开发

  1. Spring MVC是Spring体系的轻量级Web MVC框架
  2. Spring MVC的核心Controller控制器,用于处理请求,产生响应
  3. Spring MVC基于Spring IOC容器运行,所有对象被IOC管理

2.2.5. 中文乱码

Web应用的中文乱码由来：

  1. Tomcat默认使用字符集ISO-8859-1，属于西欧字符集
  2. 解决乱码的核心思路是将ISO-8859-1转换为UTF-8
  3. Controller中请求与响应都需要设置UTF-8字符集

中文乱码的配置（一般三步都需要进行设置）:

  1. Get请求乱码 - server.xml增加URIEncoding属性（在Tomcat8以后的版本已经默认为UTF-8了）
    本机server.xml的路径为：/Users/swenw/Library/Tomcat/conf/server.xml，在指定代码部分加入URIEncoding属性
  2. Post请求乱码 - web.xml配置CharacterEncodingFilter
  3. Response响应乱码 - Spring配置StringHttpMessageConverter
2.2.6. 响应输出结果

在响应中产生结果有如下两种方式：

  1. @ResponseBody - 产生响应文本
  2. ModelAndView - 利用模板引擎渲染输出
  • @ResponseBody
    • 直接产生响应体的数据,过程不涉及任何视图
    • 可产生标准字符串/JSON/XML等格式数据
    • 被StringHttpMessageConverter所影响（Response字符配置等）
  • ModelAndView
    • 是指”模型(数据)与视图(界面)”对象
    • 可将包含数据对象与模板引擎进行绑定
    • SpringMVC中默认的View是JSP,也可以配置其他模板引擎
    • mav.addObject()方法设置的属性默认存放在当前请求中
    • 默认ModelAndView使用请求转发(forward)至页面
    • 重定向使用 new ModelAndView(“redirect:/index.jsp”)

  重定向的使用场景：当当前页面与要跳转的页面没有直接联系的时候，可以使用重定向的跳转模式，假如在注册完成之后需要进行跳转到首页时

3. RESTful开发风格

  传统的MVC模式下，使用JSP渲染输出的是HTML，但是随着互联网的发展，除了像IE这种B/S结构下的传统客户端，还有像微信小程序、APP这种其它的的应用客户端，这些是不支持HTML的，这种情况该如何与后端使用HTML进行通信呢。这时全新的开发风格产生了，就是REST与RESTful
  使用RESTful开发模式，服务器返回的数据会以JSON或者XML格式进行返回，不会在数据中夹杂任何与展示有关的内容。对于这种模式还有一种专门的名词：前后端分离

RESTful开发规范：

  1. 使用URL作为用户交互入口
  2. 明确的语义规范(GET|POST|PUT|DELETE)
  3. 只返回数据(JSON|XML),不包含任何展现

3.1. RESTful注解

  1. @RestController - 创建RESTful风格控制器对象
  2. @PathVariable - 路径变量注解
  3. @GetMapping/@PostMapping - 对应请求注解

3.2. 简单请求与非简单请求

  1. 简单请求是指标准结构的HTTP请求,对应GET/POST请求
  2. 非简单请求是复杂要求的HTTP请求,指PUT/DELETE、扩展标准请求
  3. 两者最大区别是非简单请求发送前需要发送预检请求

  预检请求能够经过服务器的预先处理，将不符合规范的请求挡在外面，实际的数据并不会发送，这样就能减轻网络传输及服务器的压力了，但是如果在Spring MVC中不进行特殊处理，那么Put与Delete是无法获取到传输的数据的。因此，在web.xml中需要进行拓展：

3.4. 浏览器的同源策略

  1. 同源策略阻止从一个域加载的脚本去获取另一个域上的资源
  2. 只要协议、域名、端口有任何一个不同，都被当作是不同的域
  3. 浏览器Console看到Access-Control-Allow-Origin就代表跨域了

在HTML中允许跨域的标签：

  1. <img> - 显示远程图片
  2. <script> - 加载远程JS
  3. <link> - 加载远程CSS

解决方法：
